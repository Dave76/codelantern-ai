A2D is not vibe coding

Vibe Coding — Definition

  • Development approach where the human sets the direction and the AI handles the details
  • Emphasis on speed, experimentation, and “vibes” rather than clear structure
  • Heavy reliance on natural-language prompts with minimal scaffolding
  • Great for prototypes and quick experiments, but weaker on governance and repeatability

A2D on the spectrum

  • Traditional Development – Little or no AI assistance
  • Vibe Coding – AI drives most implementation; humans steer loosely
  • A2D – Structured collaboration between humans and AI agents, with:
    • Clear roles
    • Guardrails and governance
    • Repeatable workflows

Hiding the IP

  • Agents depend on definitions, archetypes, and workflows stored as Markdown
  • Markdown is:
    • Easy to read
    • Easy to copy
    • Hard to protect as proprietary IP
  • We need to:
    • Conceal and version these artifacts
    • Keep client copies in sync with a central source
    • Offer controlled access to paying customers

Conceptual Proposed Solution*

  • Safeguard IP behind a dedicated CodeLantern MCP server
  • Publish that MCP server through the GitHub registry
  • Issue access tokens to subscribers
  • Support multiple versions:
    • Backwards compatibility
    • Ability to evolve with MCP ecosystem changes

* This is a conceptual design; implementation details still to be validated.

Proposed Architecture

  • Work item (GitHub issue) is assigned to an agent
  • Agent opens a PR and starts a working session
  • Client org keeps agent definitions under /.github/agents
  • MCP servers involved:
    • GitHub MCP Registry
    • OpenAI Connector Registry
    • Figma / UX MCP
    • Azure MCP
    • CodeLantern MCP (hides archetype & workflow IP)
  • Agents execute workflows and submit work for human review

Workflows – Visual

  • Think of each workflow as a binder:
    • Clear entry criteria
    • Defined steps
    • Expected outputs
  • A2D is a set of binders that teams can adopt and compose

Planning Workflow

  • Uses the [planning-agent]
  • Responsibilities:
    • Enhance descriptions & acceptance criteria
    • Split large items into sub-items and tasks
    • Ask clarifying questions where needed
  • Can recommend next workflows by analyzing:
    • Repo structure
    • Existing issues
    • Nature of the work
  • Outcome: a high-quality, agent-ready backlog

Suggestion Workflow

  • Scans:
    • Current issues
    • TODOs
    • Markdown checklists
    • Test coverage gaps
  • Produces:
    • A simple work item summarizing what to work on next
  • Used by:
    • Product owners
    • Architects
  • Often powered by the same [planning-agent] under the hood

User Interface Design Workflow (Future)

  • Describe the UI that needs to be built
  • Generate UI options using project archetypes:
    • Material Design
    • MudBlazor
    • Tailwind, etc.
  • Present options and refine with stakeholders
  • Attach the chosen design to a new implementation work item

Solution Design & Analysis Workflow

  • Deep analysis across:
    • Archetypes
    • Current repository
    • Prior problem statements
  • Produces 2–3 design options with:
    • Detailed approaches
    • Code samples
    • Markdown documentation
  • Scores options and makes a recommendation
  • Also advises on timing:
    • Do it now vs. defer

Implement Workflow

  • This is the workhorse of the system
  • A [coder] agent is instructed to:
    • Be concise and efficient with changes
    • Minimize refactoring unless requested
    • Respect existing patterns and decisions
    • Write appropriate unit tests
    • Produce well-documented PRs with screenshots where helpful

Deployment Workflow

  • Uses a [deployment] agent with skills in:
    • Terraform
    • Azure CLI
    • YAML
    • GitHub Actions
  • Responsibilities:
    • Inspect solution for deployment readiness
    • Create Terraform and supporting scripts
    • Ensure environment consistency
    • Keep secrets safe and managed
    • Enable repeatable deployments

Methodology Definition

  • As important as the technology itself
  • A2D will define ceremonies similar to Agile, but with shorter cycles
  • Likely better aligned with Kanban than strict sprints
  • Daily flow typically:
    1. Plan – morning planning with agents
    2. Design – refine approaches
    3. Implement – coding agents execute
    4. Review – humans review & approve
    5. Deploy – promote safely
    6. Verify – validate outcomes
  • Then repeat

The Human Role

  • Humans remain central:
    • Provide context
    • Set direction & priorities
    • Give positive and negative feedback
    • Review and approve all changes
  • A2D is about augmenting, not replacing, the team

Risks

  • GitHub Copilot and MCP ecosystem are moving fast
  • Many features are still in preview
  • Registries and servers are unstable; things break frequently
  • GitHub keeps tight control over agents and currently:
    • Limits coding agents largely to Anthropic Sonnet
  • Licensing:
    • $15–$30 USD per user per month today
    • Unlikely to stay that low long-term
  • Agent drift:
    • Same inputs can produce slightly different outputs
    • Governance and guardrails are essential

Phased Approach

  • Start as a services company while artifacts mature
  • Package mature artifacts as a subscription offering later
  • Allow time for:
    • MCP landscape to stabilize
    • Archetypes & workflows to be battle-tested
  • Generate revenue while:
    • Building & testing artifacts
    • Proving the CodeLantern MCP concept
